Issue #308 — Root-Owned TUI Layout Budget

2026-06-13 — worktree fix/root-layout-budget-308-20260613

Summary

The root App model (tui/internal/tui/app.go) sizes every child screen by forwarding the raw terminal tea.WindowSizeMsg straight through. Child screens therefore size themselves to the full terminal height. Any persistent root-level chrome (a status bar, a warning banner) then has to be appended after the child has already rendered at full height — which pushes content off-screen or corrupts the layout.

Today the only such consumer is the startup banner (the Codex-auth warning), which View() prepended to the mail content after the mail view had already sized itself to the full terminal height. This PR builds the small, explicit layout-ownership contract the banner — and any future root chrome / status line — can consume safely. It is the foundation, not a revival of the full status-line feature from PR #241.

Non-goal: no new persistent status-bar setting or UI. Bottom chrome is an inert hook reserving zero rows until a real product surface needs it.

The contract

New file tui/internal/tui/layout.go:

type LayoutBudget struct {
    Width  int
    Height int // full terminal height

    TopChromeRows    int // rows reserved at top for root chrome
    BottomChromeRows int // rows reserved at bottom (0 today)
    ChildHeight      int // height handed to the child (clamped >= 0)
}

func (b LayoutBudget) ChildWindowSize() tea.WindowSizeMsg  // full width, reduced height

func (a App) layoutBudget() LayoutBudget   // reserve chrome rows, clamp child >= 0
func (a App) topChromeRows() int           // 1 when startupBanner != "", else 0
func (a App) bottomChromeRows() int        // always 0 (inert hook)
func (a App) topChrome() string            // renders the banner row
func (a App) composeWithChrome(child) string // stacks chrome above child content

The flow: reserve → forward → compose

StageBeforeAfter
Reserve — layoutBudget() reserves topChromeRows() + bottomChromeRows() and computes ChildHeight.
Forward (Update & sendSize) child receives raw terminal Width×Height child receives ChildWindowSize() — full width, height reduced by reserved rows
Compose (View) banner prepended to mail content after the child sized to full height (mail-view branch only) composeWithChrome() stacks root chrome above the already-reduced child content, for the whole app

Why this is the status-area foundation

Root chrome ownership is now explicit and declarative. A future persistent status line plugs in by: (1) declaring its rows in layoutBudget() (e.g. bottomChromeRows() returns 1), and (2) rendering them in a chrome helper composed by View(). The child screen automatically yields the space because every sizing path — the live resize (Update) and the route-entry seed (sendSize) — forwards the reduced budget. No screen has to know about the status area; the contract is enforced once, at the root.

Limitations & next steps

  • Bottom chrome is inert. bottomChromeRows() returns 0; it exists only as a tested hook. A real bottom status line is a follow-up PR.
  • Banner is one row. The current banner is single-line by construction (firstLine() sanitises embedded newlines elsewhere). A multi-row chrome consumer would set topChromeRows() accordingly and ensure topChrome() renders exactly that many rows.
  • Child re-clamping unchanged. Screens still re-clamp the height they receive to their own minimums (e.g. mail's viewport floor of 1). The budget only guarantees a non-negative forwarded height; per-screen minimums are out of scope.
  • No new product surface. Deliberately no status-bar setting/UI — this is the layout refactor foundation only (distinct from PR #241).
