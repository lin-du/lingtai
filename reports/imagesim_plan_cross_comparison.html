ImageSimilarity 两版方案交叉评估

对照对象：① 我先前的 HTML 研究报告；② 春水提供的 Claude code's plan.docx。结论重点不是“谁对谁错”，而是两者处在不同层级：我的报告偏研究/证据链与现有 repo 风险诊断；Claude Code plan 偏 LingTai 产品化落地与代码 scaffold。
总判断：两版最好合并，而不是二选一。我的方案适合作为“先别误工程化”的底层诊断：先稳定 ImageSimilarity 的数据契约、manifest、可复现性与证据边界。Claude Code plan 适合作为“如果要真的接进 LingTai 产品体系，应怎么做”的实现蓝图：MCP addon、watchdog、TUI 注册、forensic skill、26-code taxonomy、验证测试。合并路线应是：先 manifest/job_status/page_index，后最小 MCP；先证据工厂，后 agent 网络。

2. 我这版的优点与不足

优点

  • 对现有 CV repo 更谨慎：把 Colab/Drive 私有路径、外部 weights、ASpanFormer、requirements、CUDA hardcode、RGB/gray mismatch 等提前暴露。
  • 证据边界清楚：明确 CV/MLLM 只是线索，不应直接升级为人文结论。
  • 文章吸收较深：把 Taylor & Francis 的核心方法转成编辑痕迹 taxonomy 和 full-page/context 需求。
  • schema-first：强调 stable image_id/pair_id/hash/run metadata，这是真正让后续 agent 可引用、可审计的基础。

不足

  • 实现落点不够具体：没有给出 TUI/Kernel/MCP 具体文件、config template、测试路线。
  • 运行监控不足：只说 async/GPU jobs，没有像 Claude 那样设计 active watchdog。
  • taxonomy 不够工程化：文章导出的标签适合人文解释，但没有 closed 26-code 那样适合 schema validation。
  • 没有吸收春水后续回答：例如 crop filename 可推出 full page、Gemini 3.1 Pro、需要 scaffold Go TUI repo 等信息，我当时并不知道。

3. Claude Code plan 的优点与不足

优点

  • 产品化路径非常具体：lingtai-imagesim MCP addon、7 个 MCP tools、config、schemas、tests、TUI 注册点都有。
  • watchdog 设计好：不是“提交后等完成”，而是能发现 stall/timeout/failure 并邮件唤醒 agent，符合长 GPU job 的真实运维需求。
  • 把春水补充需求纳入：full-page mapping、Gemini 3.1 Pro、scaffold 哪些属于 Go TUI repo、哪些属于 sibling repo。
  • 验证计划完整：Go build、addon pytest、schema strict-mode、mock Gemini、fake stalled job 等都列出。

不足/风险

  • 太快进入 MCP/TUI 工程：如果 manifest/job_status/page_index 未先验证，TUI 注册会变成空壳。
  • 对 ImageSimilarity 现有风险覆盖不足：没有充分处理 RGB/gray mismatch、requirements 缺失、weights provenance、basename overwrite、full score matrix scale 等。
  • 部分假设需验证：crop 文件名是否真能稳定推出 page/bbox；aspan_breakpoint=50 是否可靠；CV source/target 与 chronology 的关系；Gemini 3.1 Pro 实际 API/成本/版权边界。
  • taxonomy 可能过度封闭：26-code validation 好，但历史图像编辑常有“低清印刷/再制传播/图注歧义”等灰区，不能把 unknown/uncertain 压扁。

4. 最推荐的合并路线

Phase 0 — 对齐事实 取 20–50 对真实样例：crop filename、full page、output JSON、match_log、OCR/caption/issue metadata。 确认 source/target/fullsource 定义、weights、ASpanFormer commit、breakpoint、Gemini/版权边界。 Phase 1 — 先做 ImageSimilarity 侧 thin scripts make_page_index.py + emit_manifest.py + job_status.json contract。 不先改 train/match/process；但要记录当前风险，并为 RGB/gray、CUDA、路径等开 issue。 Phase 2 — 最小 lingtai-imagesim addon 先只做 job_status/list_pairs/get_pair/compare_pair（mock MLLM 起步）。 submit_job/watchdog 可接 fake runner，再接真实 GPU runner。 Phase 3 — Forensic skill + 双层 taxonomy 低层 closed 26-code 便于 schema；高层 DHF 标签便于解释：republication、recaption/recontextualization、print_degradation、propaganda intent 等。 Phase 4 — TUI/Kernel 注册 只有当 addon 的 manifest/status/tool schema 跑通后，再加 AllAddons/preset template/examples。 Phase 5 — 人文报告 agent analyst avatars 按 issue/publication shard；输出 findings JSON；orchestrator 汇总 report.md/json。

5. 可以直接采纳 Claude Code plan 的部分

  • active watchdog + mailbox wake：这是长 GPU pipeline 必需。
  • job_status.json：作为 transport-agnostic heartbeat contract，很好。
  • make_page_index.py：只要 filename convention 验证成立，就能补上 full-page context。
  • compare_pair as MCP tool：比内置单图 vision 更适合 2–4 图比较。
  • strict MCP schema 注意事项：避免 top-level oneOf/enum 等 provider strict-mode 问题。
  • 测试路线：尤其 fake stalled job、mock Gemini、manifest validator。

6. 我会保留/补进 Claude plan 的部分

  • 在 design doc 的前面加一节“不要忽略现有 CV repo 的工程风险”，列出 CUDA/依赖/weights/RGB-gray/路径/scale。
  • 把 CV/MLLM outputs 定义成 leads / hypotheses，必须有 human review 字段。
  • 把 26-code taxonomy 和文章导出的 DHF taxonomy 合并：一个用于机器验证，一个用于人文解释。
  • 把 MVP 门槛设为“给定 fixture 能复现 manifest + compare_pair mock + report”，而不是“先完成 TUI 注册”。
  • 把版权/第三方 MLLM 上传图像的风险提升到 build 前置检查。

7. 一句话结论

如果春水要的是“研究判断”：看我原 HTML 更好，因为它说明为什么不能直接把 notebook 交给 agent 跑。
如果春水要的是“工程落地”：Claude Code plan 更好，因为它把 LingTai MCP/addon/watchdog/TUI/scaffold 写到了可开工粒度。
真正最强版本：用我的报告做 Phase 0/1 的风险闸门，用 Claude plan 做 Phase 2–5 的实现蓝图。
