Text output now follows API-call spacing

Small TUI fix for the ctrl+o transcript: assistant text_output entries are grouped like tool calls, so separate LLM API responses do not visually run together.

Scope
TUI mail transcript
Behavior
Blank line on new API group
Risk
Low, render-only grouping

What changed

  • MailModel.buildMessages now assigns the current api_call_id to text_output, not only tool_call/tool_result.
  • renderMessages tracks consecutive visible text_output entries and inserts one blank separator line when their API-call group changes.
  • Legacy transcript rows with no grouping metadata stay grouped, avoiding speculative spacing.
  • Focused tests cover API id derivation, separator helper behavior, and rendered blank-line output.

Before / after

Before
• [text_output] first response
• [text_output] second response

Different LLM responses could appear as one dense block.

After
• [text_output] first response

• [text_output] second response

A new API response gets the same visible separation as tool-call groups.

Why this is the right layer

The kernel already writes hidden llm_response markers with API-call ids. The TUI already derives group ids from those markers for tool display. The missing piece was simply letting text_output participate in the same grouping contract.
