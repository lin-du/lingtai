Issue #179 — Evidence-verification gate before academic draft generation

Docs/skill change Branch: feat/academic-evidence-gate Source: tui/internal/preset/skills/swiss-knife/reference/academic-research/

TL;DR

LingTai already ships an academic-research skill that can fetch and verify papers (DOI/arXiv/PMID, Crossref/OpenAlex/Unpaywall/Europe PMC). But the safe workflow was advisory: when a user asked to "go fast" or for a "readable draft," the model could slide straight from reference discovery into confident prose, flattening DOI-verified papers, preprints, and search leads into one even evidence layer that reads as submission-ready.

This change adds an evidence-verification gate: a new reference that requires a verified literature matrix before any citation-bearing draft, keeps source-reliability tiers (A/B/C) separate in the prose, and routes academic-writing situations into academic-research before fluent manuscript drafting. The skill index, escape-hatch paths, and Known caveats now route to it.

Scope: documentation/skill content only. No Go runtime behavior changes; one bundling test gains an assertion for the new file. No config, release, or secrets touched.

Baseline (the incident)

A user asked for a fast HCI paper based on a custom break-reminder skill. The agent produced a readable draft and a literature matrix — but mixed DOI-verified HCI / digital-wellbeing sources with preprints and search leads at the same evidence level.

The draft was not pure fabrication, but the evidence layers were flattened: a non-expert reader would believe every reference was equally submission-ready. The user caught it and noted this would likely draw reviewer rejection and waste more time than verifying first. The correction worked — but only after the user intervened.

Root cause: the model over-amplified "fast" into "skip verification," prioritized fluent prose over source checking, and hid uncertainty behind polished writing. The capability to verify existed; nothing routed the model through it before drafting.

What changed

1. New evidence-verification-gate reference

reference/evidence-verification-gate.md — the verify-before-drafting gate. It covers: why the gate exists (the four LLM failure tendencies + the incident); detection (the author–year / References catch-all, plus high-confidence and Chinese/English human-pushback triggers, and a medium-confidence question instead of hard blocking); the verified-literature-matrix schema (title, authors, year, venue, identifier, verified_by, evidence_class, claim-it-can / cannot-support, notes); A/B/C evidence tiers; the "go fast = reduce scope, not verification" stance; the matrix-before-draft artifact convention (with matrix path in the draft header to survive context loss); a pre-draft lint; and caveats (latency, non-English venues, verified_by: none never silently A-tier, guard fatigue).

2. Skill routing — index, escape hatches, caveats

SKILL.md gains a prominent "Before drafting citation-bearing academic writing" banner above the escape-hatch block, two new escape-hatch quick-paths ("I'm about to write a paper / references / related-work section" and "the user wants a fast / readable draft"), a Standalone references entry, and a Known caveat bullet that routes drafting-without-verification to the new gate.

3. Bundling test assertion

swiss_knife_populate_test.go now asserts the new reference is extracted to disk, guarding against a future move silently dropping it.

Evidence tiers (kept separate in the prose)

Tierevidence_classWhat it may do in the draft
A peer-reviewed / conference paper (DOI- or venue-verified) May support main claims as a formal citation.
B preprint / posted content (arXiv, bioRxiv, SSRN, online-first) Must be labeled preliminary. Never silently A-tier.
C search lead / news / blog / unverified result Lead only — not evidentiary support, never a formal citation.
Rule: verified_by: none, future-year, or online-first items must never silently become A-tier evidence.

Risks

  • Low blast radius. Content-only change to a bundled skill; no Go logic, schema, or migration touched.
  • The gate is a documentation guard, not a hard runtime block — by design. The issue's own analysis warns against hard-blocking (guard fatigue, false positives), so the gate routes and prompts rather than refusing.
  • The original incident's specifics are used only as one anonymized illustrative example, not as instructions.
  • Embedding is via //go:embed skills (recursive), so the new file ships automatically; the test assertion guards a future move from silently dropping it.
