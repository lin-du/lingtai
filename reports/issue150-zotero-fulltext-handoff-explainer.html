Issue #150 — Zotero institutional full-text handoff workflow

Docs/skill change Branch: docs/issue-150-zotero-fulltext-handoff-20260609 Source: tui/internal/preset/skills/swiss-knife/reference/academic-research/

TL;DR

The academic-research skill walks an open-access → authorized-publisher → LibGen ladder for "get me this paper." When OA and the authorized-publisher HTTP probe both miss for a whole batch of papers, there was no documented agent-native route to use the one tool that usually still works: Zotero Desktop's UI Find Full Text (查找全文), which resolves full text through the user's institutional access.

The catch: an agent cannot reliably drive Zotero's Find Full Text. Its Local API item routes are read-only, the Connector's attachment resolver is narrower than the UI path, and macOS TCC/Accessibility blocks AppleScript UI automation. This change documents the practical, in-policy answer: a human-in-the-loop handoff — the agent stages a tagged batch into Zotero, the human clicks Find Full Text, and the agent harvests the returned PDFs back with provenance.

Scope: documentation/skill content only. One new reference file plus cross-link routing through the existing ladder and a self-contained explainer. No code, no UI automation, no credential handling, no paywall bypass.

Baseline (the incident)

During a production HelioSI full-text acquisition sprint, OA fetchers and the authorized-publisher probe left a large tail of fetch_failed records. Zotero Desktop 9.0.3, on an institutional network, could resolve many of them via UI Find Full Text — but the agent had no robust way to invoke that path and discovered several non-obvious boundaries mid-sprint.

The agent burned many turns probing the Local API (POST → 400 "Endpoint does not support method"), the Connector resolver (500 Failed to save an attachment on samples the UI later resolved), and macOS AppleScript automation (-1719 "not allowed assistive access", -1743 "Not authorized to send Apple events to System Events"). It eventually fell back to: bulk-stage 5222 failed records into Zotero via /connector/saveItems under tag HelioSI bulk fulltext queue 20260521, have a human run Find Full Text, then harvest the new PDFs. That worked — but it was undocumented.

Root cause: the missing, documented route cost substantial time and affects any large authorized-acquisition workflow that relies on institutional Zotero access. The fix is to write the working route down and route to it from the skill's ladder — not to automate the human's click.

Why this is a handoff, not an agent tool

These boundaries (mapped against Zotero Desktop 9.0.3, local API at 127.0.0.1:23119) are exactly why the workflow keeps a human in the loop:

SurfaceWhat it doesWhy it isn't Find Full Text
/api/users/0/items (Local API) Read items (GET). POST → 400 "Endpoint does not support method". Cannot create/mutate items.
/connector/saveItems Create item metadata from DOI/URL, with tags. Creates the row only — no full text. This is the staging primitive we do use.
/connector/saveAttachmentFromResolver getFileResolvers(item, ['oa','custom'], true). Only oa+custom — narrower than the UI; returned 500 on samples the UI resolved.
UI Find Full Text (zoteroPane.js) Zotero.Attachments.addAvailableFiles(getSelectedItems()); resolvers include doi,url,oa,custom. The broad path that actually finds PDFs — reachable only from the UI. No HTTP endpoint exposes it.
macOS AppleScript automation (attempted) TCC blocks it: -1719 / -1743. Out of policy and unreliable — do not pursue.
A human clicking one menu item is faster, more robust, and within policy than fighting TCC/Accessibility. The skill stages and harvests around that click.

The handoff workflow

  1. Generate a handoff manifest. One inspectable CSV/JSON row per already-failed paper (local_id, doi/url, title, batch_tag, staged, harvested). This manifest — not Zotero's DB — is the source of truth for what was asked for and what came back, and it bounds the batch the human is asked to act on.
  2. Stage metadata into Zotero (agent). One item per row via /connector/saveItems — the only write path that works — each tagged with a unique dated batch tag (LingTai bulk fulltext queue 20260609) plus a provenance note. Metadata only; idempotent (never re-stage a row already staged).
  3. Hand off to the human (the part the agent cannot do). Ask the user, in Zotero Desktop, to filter by the batch tag, select all, right-click → Find Full Text, and confirm when done. The agent does not click these menus.
  4. Harvest returned PDFs (agent). Read the tagged items back via the Local API (GET), find new PDF child attachments in Zotero storage, and for each: validate %PDF- magic bytes + size, copy (not move) atomically into papers/{slug}/paper.pdf, and record provenance (tier: zotero_institutional_handoff, resolved_by: human Find Full Text, batch tag, Zotero keys). Reconcile against the manifest and surface what did not come back. 
{
  "status": "ok",
  "tier": "zotero_institutional_handoff",
  "route": "zotero_find_full_text_human",
  "provenance": {
    "doi": "10.1007/s11214-020-00743-1",
    "batch_tag": "LingTai bulk fulltext queue 20260609",
    "zotero_item_key": "ABCD1234",
    "resolved_by": "human Find Full Text (Zotero Desktop institutional access)",
    "content_type": "application/pdf",
    "bytes": 4123456
  }
}

No cookie, session token, proxy auth, Authorization, or Set-Cookie value is ever read from Zotero or written into provenance. The honest attribution is "resolved by a human running Find Full Text."

Where it sits in the ladder

arXiv → Unpaywall → Europe PMC → CORE → publisher-extract (Tier 5)
                                          → authorized-publisher (Tier 5b)
                                          → Zotero institutional handoff (Tier 6a)   ← new
                                          → LibGen (last resort)
  • vs. Tier 5b (authorized-publisher HTTP probe): 5b is a single automated GET of the publisher's own PDF. Tier 6a engages Zotero's broad resolver set through the human, recovering papers 5b's single GET cannot. Try 5b first; hand off the leftovers here.
  • vs. LibGen: LibGen is a shadow library of unclear copyright status. Tier 6a is the licensed publisher copy via the user's institution. Always prefer 6a.
  • Batch-shaped: worth it only for a list of failures — the human cost is "select N, click once." For a single DOI, use 5b.

Hard boundaries

This tier must never:

  • drive the Zotero UI by force — no AppleScript/Accessibility/Apple-events menu traversal, no synthetic clicks, no TCC/Automation bypass;
  • bypass or solve a paywall, CAPTCHA, or anti-bot challenge;
  • capture, store, replay, guess, or forge credentials, cookies, session tokens, or proxy auth — Zotero's institutional access is the user's session, owned at the OS/network layer;
  • scrape a hidden/authenticated session the user has not themselves established;
  • fall through to LibGen/Sci-Hub automatically — that is a different, stricter tier.

If the human declines, or Zotero is off-network, the tier simply misses and the ladder falls through. It is a way to use access the user already has, with the user in the loop — not a way to obtain access.
