ImageSimilarity → 灵台式 Agentic Network：CV pipeline 嵌入方案

研究对象：https://github.com/JordanLow/ImageSimilarity；本地 clone commit 71266e0。方法：静态代码/Notebook 审阅 + Taylor & Francis 文章阅读 + 两支分神 + claude -p --model opus 独立审阅。未训练、未下载大模型、未运行 GPU pipeline。
总判断：这个 repo 现在不是可以直接“插进灵台”的稳定组件，而是一个 Colab/Drive 驱动的研究型 notebook pipeline。它最适合被改造成一个 schema-first evidence factory（证据工厂）：CV 负责发现候选 image pairs，几何验证给出可度量的对应证据，pair curator 形成审阅队列，MLLM agent 再对原始整页/裁剪图做编辑痕迹标注，最后由人文研究 agent 写解释性报告。关键不是让一个 agent “跑 notebook”，而是把 notebook 拆成可审计、可复现、可排队、可引用的工具链。

1. 现在的 CV pipeline 实际在做什么

Google Drive / Colab → stage source/target crop zips 到 /content/image_similarity_work → train.py 训练自监督 contrastive image-similarity head（可选/离线） → match.py 对 source×target 做 embedding cosine top-k retrieval → process_images.py 用 ASpanFormer + RANSAC 做几何验证 → output/*.json、visualizations/*/match_log.json、weights/、zip 回 Drive
阶段当前实现主要输入/输出可 agent 化的接口
数据 stagingrun.ipynb 从 Drive 复制 workingsourcecrops.zip、working_targetcrops.zip 到本地 Colab 并解压输入：Drive zip / private paths；输出：source/target crop dirsingest(source_dir/archive, target_dir/archive) → manifest/images.jsonl
训练train.py + DataLoader.py：把 source+target 混合，用原图/增强图构造自监督正对；冻结 backbone，只训练 FC/projection head输出：weights/best.pt（主要是 head 权重）先作为离线 model-versioning，不放在默认 pipeline 每次运行中

2. 不能直接产品化的关键风险

Colab/Drive 强耦合

run.ipynb 写死 /content/、google.colab、Drive 路径；scripts 从 Drive 而非 git checkout 运行，可能与 repo 版本漂移。

依赖与权重不完整

requirements.txt 只有 torch/torchvision/pillow/tqdm/numpy；ASpanFormer、OpenCV、Lightning、Albumentations 等不在 repo 管理；DINO/ASpanFormer 权重靠私有路径。

CUDA 硬依赖

DataLoader、match.py、process_images.py 多处直接 .cuda() 或 device='cuda'；没有 CPU fallback，也没有清晰的早期失败。

训练/推理预处理不一致

DataLoader.readImg() 保持 RGB；match.py.readImg() 会转成 3-channel grayscale。这可能是 train/test distribution shift，必须确认是否有意为之。

输出契约脆弱

per-source JSON 用运行时绝对路径、basename 命名；重复 basename 会覆盖；match_log.json 只存 target path→inlier count，丢失 similarity/rank/threshold/model metadata。

Notebook 命令漂移

Notebook 中有 --top_x_matches 例子，但当前 process_images.py 无此参数；epochs/threshold 等也有多处不同版本。

3. 放进灵台/agentic network 的推荐拓扑

Orchestrator（主控） → image-ingestion-indexing agent：解包、manifest、hash、页/图元数据 → cv-retrieval GPU service/agent：embedding + top-k candidates → geometric-verifier GPU service/agent：ASpanFormer/RANSAC 验证 → pair-curator agent：合并证据、去重、分级、形成 review queue → mllm-edit-tagger agent：整页+裁剪图比较，输出编辑痕迹 tags → humanities-report-writer agent：结合出版物/图注/上下文写人文报告 → provenance-audit-keeper：run_id、参数、模型、hash、日志、schema validation
组件在灵台里适合的形态理由
Pipeline 使用说明Skill：image-similarity-pipeline让任何研究 agent 知道何时调用、输入格式、如何解释输出、哪些结论只是线索。
稳定 CV 执行接口MCP 或本地 API serviceCV retrieval/verification 是 typed tool；避免每个 agent 直接 shell notebook。
长 GPU 任务service 背后的 async bash/job queueembedding、ASpanFormer、训练都可能很久，应返回 job_id/run_id 而不是阻塞 agent。
单次项目编排cv_pipeline_keeper daemon/avatar拥有 run workspace、轮询 job、写 audit/events.jsonl、把 artifact paths 回给研究 agent。
模型卡/阈值/失败模式Knowledge记录 project-private 的模型版本、权重 hash、阈值经验、Jordan/春水确认过的设定。
候选/验证/MLLM 输出Artifacts研究报告引用 artifact id/path/hash，而不是把大 JSON 塞进对话。

4. 数据契约：把“图片路径”升级为“可引用证据”

最小需要以下 JSON/JSONL 契约。核心原则：每个 pair 有稳定 pair_id；每张图有 image_id/sha256；每个结论都能追溯到 run、模型、参数和 artifact。

audit/run.json
manifest/images.jsonl
retrieval/candidates.jsonl
verification/pairs.jsonl
curation/curated_pairs.jsonl
curation/review_queue.jsonl
mllm/edit_tags.jsonl
reports/findings.jsonl

RetrievalCandidate 需要包含

  • run_id, pair_id
  • source/target image id + path
  • rank, similarity_score
  • model_definition / weights hash / preprocessing version
  • self-match suppression 与错误状态

GeometricVerification 需要包含

  • ASpanFormer config/weights hash
  • raw_keypoints, inlier_count
  • RANSAC method / reproj threshold / breakpoint
  • high/low/failed 状态
  • 可选 visualization_ref、fundamental matrix / transform hint

5. MLLM 编辑痕迹 tagger：为什么必须看原始整页

春水给的 Taylor & Francis 文章（From information to metaphor: Tracking photographic editing in Chinese wartime magazines through Digital Historical Forensics）正好说明：CV 找 pair 只是第一层，真正的人文问题在“图像怎样被裁剪、重配图注、再发表、修饰、误导性重用”。文章本身也采用 YOLOv7 + ViT/EfficientNetv2/Swin top-15 retrieval（报告 77.8% top-15 accuracy）来辅助追踪，但最终解释依赖图注、版面、出版语境和传播路径。

因此 MLLM agent 不应只看 YOLO crop；它应接收：

  • source/target 的原始整页图（带 crop bbox overlay）；
  • 对应 crop 局部图；
  • CV metadata：similarity score、rank、ASpanFormer inliers、可视化；
  • OCR/图注/出版物 metadata（如果有）；
  • 严格 JSON 输出 schema，区分“可见视觉事实”和“历史解释假设”。

建议 controlled vocabulary

crop / reframepartial_overlapmirror / fliprotation / scalerecaptioncaption_removed / addedlayout_changeretouch / erased_objectdrawn_additionmontage_or_compositerepublication / circulationprint_degradation / detail_suppressionnon_matchuncertain 

{
  "pair_id": "...",
  "relationship": {
    "is_same_base_image": true,
    "relationship_type": "derivative | republication | non_match | uncertain",
    "confidence": 0.86
  },
  "edit_tags": [
    {"tag":"crop", "confidence":0.92, "evidence":"target omits left margin and lower foreground", "visual_basis":"full_page+crop"},
    {"tag":"recaption", "confidence":0.81, "evidence":"caption text differs between publication contexts", "metadata_needed":true}
  ],
  "negative_tags": ["mirror"],
  "needs_human_review": true,
  "warning": "Do not claim publication history unless supported by metadata."
}

6. MVP 路线：先别重写成巨大系统，先把证据链跑通

阶段交付物目的
0. 确认输入source/target/full-page 的定义；weights；ASpanFormer 版本；threshold避免在错误数据假设上工程化。
1. 包一层 legacy runner保留 match.py/process_images.py，外面加 run_id/workspace/audit最快把 notebook 从 Colab cell 变成可调用 pipeline。
2. Manifest + converterimages.jsonl、legacy JSON → candidates.jsonl、pairs.jsonl建立稳定 pair_id 和可追溯 artifact。
3. Curation policydefault_curation.yml + review_queue.jsonl决定哪些 pair 送 MLLM/人工，而不是全量烧 API。
4. MLLM prototype对 10–50 对 pair 生成 mllm/edit_tags.jsonl + 小报告验证 crop/mirror/recaption/retouch/republication schema 是否可用。
5. 灵台集成Skill + cv_pipeline_keeper daemon/avatar；后续再 MCP让其他人文 agent 能请求 run、读取 artifact、写报告。

7. 应先问 Jordan/春水确认的问题

  1. workingsourcecrops、working_targetcrops、fullsource 的 canonical 含义是什么？fullsource 是否应该是原始整页，而不是当前同 source crops？
  2. Notebook 里把 target crops 复制进 source crops 是正式流程还是一次性 workaround？
  3. 生产 matching 应使用哪个 checkpoint：当前训练出的 weights/best.pt、Drive 的 DINO.pt，还是某个 Working_DINO 权重？
  4. ml-aspanformer 是否是 fork？应记录哪个 repo/commit/weights hash？
  5. BREAKPOINT_VALUE=50/75 是如何校准的？是否随数据集变化？
  6. match.py 的 grayscale 推理是否有意？如果不是，应统一 train/inference preprocessing。
  7. 目标数据规模是多少（source 数、target 数、topk）？这决定 full score matrix 是否还能用，或必须转 FAISS/vector index。
  8. recaption/republication 所需的 OCR、期刊、日期、页码 metadata 现在在哪里？
